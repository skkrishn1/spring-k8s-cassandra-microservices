P1
Null product returns HTTP 200 instead of 404 — controller and test both wrong
ProductController.findProductsByNameAndId() called ResponseEntity.ok(null) when the service returned null for a missing product. The corresponding test shouldReturnNotFoundWhenProductNotFoundByNameAndId asserted isOk(), masking the bug. Separately, ProductDao.findByNameAndId() called row.getString() on the result of rs.one() without a null check, causing a NullPointerException when no row was found.
Critical Fixed blind + edge ProductController.java:26–29 ProductDao.java:82–86 ProductControllerTest.java:126–136
Fix: Added null check in ProductDao.findByNameAndId() returning null when rs.one() returns null. Updated ProductController to return ResponseEntity.notFound().build() when product is null. Updated test assertion to isNotFound().
P2
Test named "shouldReturnBadRequest" asserts 200 OK — misleading name, no validation guard
The test sent {"invalidField":"value"} (unknown fields, all Product properties deserialize to null), yet asserted isOk() while the method name claimed bad-request behavior. This validated the absence of input validation as if it were correct behavior.
High Fixed blind + edge ProductControllerTest.java:191–202
Fix: Renamed test to shouldAcceptRequestWithUnknownFieldsIgnoredByJackson(). Added clarifying comment noting that proper input validation (@Valid + @NotNull/@NotBlank) is deferred (D1).
P3
Logback 1.4.14 incompatible with Spring Boot 2.3.0 (SLF4J version conflict)
Logback 1.4.x requires SLF4J 2.x. Spring Boot 2.3.0 depends on SLF4J 1.7.x. Overriding <logback.version> to 1.4.14 in pom.xml would cause NoSuchMethodError or IncompatibleClassChangeError at runtime.
High Fixed blind pom.xml:38
Fix: Reverted <logback.version> from 1.4.14 back to 1.2.3 (Spring Boot 2.3.0 managed version).
P4
Stray XML artifacts corrupt 3 SKILL.md files

Medium Fixed blind kubernetes-deployment/SKILL.md:207–208 spring-boot-code-review/SKILL.md:69–70 spring-boot-unit-tests/SKILL.md:81–82
Fix: Removed all stray XML tags from the end of all three SKILL.md files.
P5
BigDecimal price compared as double in JSON path — fragile assertion
jsonPath("$.price", is(999.99)) uses a double literal matcher. BigDecimal.valueOf(999.99) may serialize with varying precision and 999.99 is not exactly representable as a IEEE 754 double, making this comparison fragile across Jackson versions or serialization settings.
Low Fixed blind ProductControllerTest.java:120, 147
Fix: Changed to closeTo(999.99, 0.001) (Hamcrest), already covered by existing import static org.hamcrest.Matchers.*.
